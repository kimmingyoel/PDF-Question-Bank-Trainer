### 핵심 파일 구조
- backend/app 디렉터리를 중심으로 FastAPI 애플리케이션이 구성되어 있다.
- main.py는 애플리케이션 진입점이며, routers 디렉터리에는 업로드, 문제 관리, 퀴즈, 북마크 관련 API가 기능별로 분리되어 있다.
- services 디렉터리에는 PDF/DOCX 파싱 로직과 LLM을 활용한 문제 생성 로직이 포함되어 있다.
- models.py에는 문제 세트, 문제, 선택지, 북마크, 풀이 기록에 대한 ORM 모델이 정의되어 있다.

- 프론트엔드는 src/pages 단위로 화면이 구성되어 있으며, Upload, Question List, Quiz 페이지가 핵심 흐름을 담당한다.
- API 통신은 별도의 client와 커스텀 훅으로 분리되어 있어 화면 로직과 데이터 처리 로직이 명확히 나뉘어 있다.

### 데이터 명세
- 문제는 QuestionSet 단위로 관리되며, 하나의 QuestionSet에는 여러 개의 문제가 포함된다.
- 객관식 문제는 질문 본문, 선택지 목록, 정답, 해설로 구성되고, 주관식 문제는 선택지 없이 정답 텍스트를 가진다.
- 모든 데이터는 JSON 형태로 프론트엔드와 백엔드 간에 주고받으며, 데이터베이스에는 문제 풀이 이력과 북마크 여부도 함께 저장된다.
- 현재 구조상 사용자 개념은 명시적으로 분리되어 있지 않으며 단일 사용자 사용을 전제로 한다.

### 기술적 제약 사항
- 문제 추출 로직은 정해진 형식(번호, 선택지 표기, ‘정답’ 키워드 등)에 크게 의존한다. 문서 형식이 일정하지 않은 경우 일부 문제가 누락되거나 하나의 문제로 병합될 수 있다.
- AI 문제 생성은 로컬 LLM 실행 환경을 필요로 하므로, 사용자의 기기 사양에 따라 성능 차이가 발생할 수 있다.
- 인증 및 권한 관리가 없어 다중 사용자 환경에는 적합하지 않다.

### 개선 사항
- LLM을 활용한 문제 추출 로직을 정규식 기반 파싱의 보조 수단으로 적극 활용하면 다양한 문서 형식을 더 안정적으로 처리할 수 있다.
- 사용자 로그인이나 회원가입 기능을 추가해 문제, 북마크, 풀이 이력을 사용자별로 관리할 수 있게 하면 좋을 것 같다.
- 문제 수정 및 삭제 기능이 UI에 없어 잘못 추출된 문제를 정정하기 어렵다. 문제 편집 기능이 있으면 좋을 것 같다.
